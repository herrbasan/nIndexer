## Fix 1 — search_keyword content search was fake (CRITICAL)

**Problem:** `SimpleMetadataStore.searchKeyword()` only searched file paths. The `searchContent` parameter was accepted but completely ignored. Tool description said "Searches file paths AND content" — this was false.

**Fix:** `indexing.service.js:529-584` — `searchKeyword()` now delegates to `grepSearcher.grep()` with `regex: false` when `searchContent=true`, then merges path results with content results. Deduplicates by path, adds `contentMatches: {line, content}` metadata to results.

**How to test:**
```
1. Start the server: node src/server.js
2. Index a codebase (or use an existing one)
3. Call search_keyword with a function name that exists in code but NOT in any file path:
   search_keyword({ codebase: "test", query: "routeToolCall", searchContent: true })
   Expected: Returns results with contentMatches showing the line where "routeToolCall" appears
4. Verify with searchContent: false:
   search_keyword({ codebase: "test", query: "routeToolCall", searchContent: false })
   Expected: Returns 0 results (only path search)
```

## Fix 2 — Noise files polluting semantic search (CRITICAL)

**Problem:** Binary files, LOCK files, log files, .ndb segments were all getting embedded into vectors. These created "noise vectors" with broadly applicable embeddings that matched every query. The same 5 noise files appeared at top of EVERY semantic query.

**Fix:** Two parts:
1. `indexer.service.js:40-41` — Added `.ndb`, `.reg`, `.wal`, `.idx` to `BINARY_EXTENSIONS` set so they're detected as binary during file walking.
2. `indexer.service.js:181-185` — After Phase 1 parsing, splits files into `embeddableFiles` (content exists AND ≥50 chars) and `noiseFiles`. Phase 2 only embeds `embeddableFiles`. Noise files are stored in metadata only (no vector) in a post-Phase-3 step at line 301-312.

**How to test:**
```
1. Delete and re-index a codebase that has _Archive/, logs/, .ndb files
2. Check console output for: "Noise filter: N files skipped from embedding"
3. Run search_semantic for a general query:
   search_semantic({ codebase: "test", query: "how does authentication work" })
   Expected: NO .log, .lock, .ndb, _Archive files in top results
4. Verify noise files are still in metadata (path-only search still works):
   search_keyword({ codebase: "test", query: ".ndb", searchContent: false })
   Expected: Returns the .ndb files (metadata-only, no vectors)
```

## Fix 3 — Auto-run analyze_codebase after indexing (MAJOR)

**Problem:** `get_codebase_description` and `get_prioritized_files` returned "No analysis available" for all codebases because `analyze_codebase` was never called automatically.

**Fix:** `indexing.service.js:336-338` — After successful `indexCodebase()`, calls `this.analyzeCodebase({name})` asynchronously (fire-and-forget with `.catch()`). This means `get_codebase_description` and `get_prioritized_files` will work immediately after indexing.

**How to test:**
```
1. Delete and re-index a codebase
2. Immediately call get_codebase_description:

   Expected: hasAnalysis: true (may take a few seconds after indexing completes)
3. Call get_prioritized_files:
   get_prioritized_files({ name: "test" })
   Expected: Returns files split into high/medium/low (not all "low")
```

## Fix 4 — Stale ignored files not cleaned on refresh (MAJOR)

**Problem:** `_Archive` directory was indexed despite being in `ignorePatterns`. Ignore patterns added after initial indexing never removed already-indexed files. Incremental refresh only removed files that disappeared from disk.

**Fix:** `maintenance.service.js:154` — During `checkAndRefresh()`, the `toDelete` loop now also checks `this._shouldIgnore(path, false)`. Any indexed file matching current ignore patterns gets removed from the index.

**How to test:**
```
1. Have a codebase that was indexed with _Archive/ files in it
2. Verify _Archive files are in the index:
   grep_codebase({ codebase: "test", pattern: "_Archive", regex: true, limit: 5 })
3. Trigger a maintenance refresh:
   refresh_codebase({ name: "test" })
4. After refresh, check _Archive files are gone:
   search_keyword({ codebase: "test", query: "_Archive" })
   Expected: 0 results (files removed during refresh)
```

The `search()` method also had a bug where `!r.language` (undefined) always passed the filter, but actual language values could mismatch.

   - **Problem:** `get_codebase_description` heavily relies on the most frequent file extension to classify a project. For example, `nui_wc2` is classified as "A webp project" instead of a UI project, simply because it contains a lot of `.webp` files. `nLogger` is labeled "A md project" because it has several `.md` files.
   - **Improvement:** Update `project-analyzer.service.js` to weigh certain structural files (like `package.json`, `.cs`, `.js`, etc.) much heavier than media or documentation files like `.webp` or `.md`.

   - **Problem:** When searching for concepts like "server initialization" using semantic search, `.gitignore` files are returning high in the results (e.g. 4th position in `llm_gateway`). 
   - **Improvement:** Expand the noise filter in `indexer.service.js` (Fix 2) to include purely declarative files like `.gitignore`, `.gitmodules`, and other config files unless they clearly belong to a search scope. They shouldn't be fully vectorized as general code structure.

fileInfo.language?.toLowerCase() !== filter.language.toLowerCase()

## Fix 6 — Grep cache fingerprint used wrong path (MINOR)

**Problem:** `grep.service.js:27` — `path.join(sourceDir, '..', 'manifest.json')` went UP from the source directory to find a manifest that doesn't exist there. The manifest is in the codebase's metadata directory (`data/codebases/<name>/manifest.json`), not relative to the source.

**Fix:** `grep.service.js:25-32` — Removed the manifest lookup entirely. Now uses `fs.stat(sourceDir).mtimeMs` as the fingerprint. This is simpler and correct — if files in the source directory change, its mtime changes.

**How to test:**
```
1. Run the same grep query twice:
   grep_codebase({ codebase: "test", pattern: "function", regex: true, limit: 5 })
   grep_codebase({ codebase: "test", pattern: "function", regex: true, limit: 5 })
   Expected: Second call is faster (cache hit)
2. Modify a file in the source directory
3. Run grep again — should get fresh results (cache invalidated)
```
