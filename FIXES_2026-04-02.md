## Fix 1 — search_keyword content search was fake (CRITICAL)

**Problem:** `SimpleMetadataStore.searchKeyword()` only searched file paths. The `searchContent` parameter was accepted but completely ignored. Tool description said "Searches file paths AND content" — this was false.

**Fix:** `indexing.service.js:529-584` — `searchKeyword()` now delegates to `grepSearcher.grep()` with `regex: false` when `searchContent=true`, then merges path results with content results. Deduplicates by path, adds `contentMatches: {line, content}` metadata to results.

**How to test:**
```
1. Start the server: node src/server.js
2. Index a codebase (or use an existing one)
3. Call search_keyword with a function name that exists in code but NOT in any file path:
   search_keyword({ codebase: "test", query: "routeToolCall", searchContent: true })
   Expected: Returns results with contentMatches showing the line where "routeToolCall" appears
4. Verify with searchContent: false:
   search_keyword({ codebase: "test", query: "routeToolCall", searchContent: false })
   Expected: Returns 0 results (only path search)
```

## Fix 2 — Noise files polluting semantic search (CRITICAL)

**Problem:** Binary files, LOCK files, log files, .ndb segments were all getting embedded into vectors. These created "noise vectors" with broadly applicable embeddings that matched every query. The same 5 noise files appeared at top of EVERY semantic query.

**Fix:** Two parts:
1. `indexer.service.js:40-41` — Added `.ndb`, `.reg`, `.wal`, `.idx` to `BINARY_EXTENSIONS` set so they're detected as binary during file walking.
2. `indexer.service.js:181-185` — After Phase 1 parsing, splits files into `embeddableFiles` (content exists AND ≥50 chars) and `noiseFiles`. Phase 2 only embeds `embeddableFiles`. Noise files are stored in metadata only (no vector) in a post-Phase-3 step at line 301-312.

**How to test:**
```
1. Delete and re-index a codebase that has _Archive/, logs/, .ndb files
2. Check console output for: "Noise filter: N files skipped from embedding"
3. Run search_semantic for a general query:
   search_semantic({ codebase: "test", query: "how does authentication work" })
   Expected: NO .log, .lock, .ndb, _Archive files in top results
4. Verify noise files are still in metadata (path-only search still works):
   search_keyword({ codebase: "test", query: ".ndb", searchContent: false })
   Expected: Returns the .ndb files (metadata-only, no vectors)
```

## Fix 6 — Grep cache fingerprint used wrong path (MINOR)

**Problem:** `grep.service.js:27` — `path.join(sourceDir, '..', 'manifest.json')` went UP from the source directory to find a manifest that doesn't exist there. The manifest is in the codebase's metadata directory (`data/codebases/<name>/manifest.json`), not relative to the source.

**Fix:** `grep.service.js:25-32` — Removed the manifest lookup entirely. Now uses `fs.stat(sourceDir).mtimeMs` as the fingerprint. This is simpler and correct — if files in the source directory change, its mtime changes.

**How to test:**

## Fix 7 — search_keyword fails for short queries (< 5 chars) — 2026-04-05

**Problem:** `search_keyword` returns empty results for short search terms like "napi" (4 chars), "os" (2 chars), "fs" (2 chars), while longer terms like "ffmpeg" (7 chars) work fine. The issue affects common short identifiers in codebases.

**Symptoms:**
| Query | Length | Result |
|-------|--------|--------|
| `napi` | 4 | ❌ Empty |
| `napi h` | 6 | ❌ Empty |
| `include napi.h` | 14 | ❌ Empty |
| `ffmpeg` | 7 | ✅ Works |
| `node-addon-api` | 14 | ✅ Works |

**Root Cause Analysis:**
1. `metadata.searchKeyword()` only searches file **paths**, not content — short terms like "napi" rarely appear in file paths
2. Content search delegates to `grepSearcher.grep()` which works correctly for short terms
3. **Empty catch block** at `indexing.service.js:544` silently swallows grep errors, making debugging impossible
4. No logging to indicate whether grep is being called or what results it returns

**Investigation:**
- ripgrep finds "napi" instantly when run directly
- `grep_codebase` works fine for short terms (uses same grepSearcher, but no try-catch)
- nVDB vector database is NOT involved — it's only for semantic search, not keyword search
- No minimum token length filtering in nVDB or grep service
- Issue is **silent failure** + **lack of visibility** into what's happening

**Fix:** `indexing.service.js:529-591` — Enhanced `searchKeyword()` with:
1. **Error logging** — logs grep errors with source path existence check
2. **Execution logging** — logs query, source path, path results count, content results count, final results count
3. **Fallback mechanism** — if both path and content results are empty, tries grep again with `regex: true` mode as a fallback
4. **Better diagnostics** — helps identify if source paths are inaccessible or if grep is failing for other reasons

**Code Changes:**
```javascript
async searchKeyword({ codebase, query, limit = 20, searchContent = true, filter }) {
  const { metadata, source } = await this._getCodebase(codebase);
  
  console.log(`[Indexing:searchKeyword] Searching "${query}" in ${codebase} (source: ${source})`);
  
  let pathResults = await metadata.searchKeyword(query, limit * 2);
  console.log(`[Indexing:searchKeyword] Path results: ${pathResults.length}`);
  
  let contentResults = [];
  if (searchContent) {
    try {
      contentResults = await this.grepSearcher.grep(source, query, {
        regex: false,
        limit: limit * 2,
        maxMatchesPerFile: 1,
        caseSensitive: false,
        excludeExtensions: [...BINARY_EXTENSIONS]
      });
      console.log(`[Indexing:searchKeyword] Content results: ${contentResults.length}`);
    } catch (err) {
      console.error(`[Indexing:searchKeyword] grep error for query "${query}":`, err.message);
      const sourceExists = await fs.access(source).then(() => true).catch(() => false);
      console.error(`[Indexing:searchKeyword] Source path: ${source}, exists: ${sourceExists}`);
    }
  }
  
  // ... merging logic ...
  
  // Fallback: if no results, try regex mode
  if (results.length === 0 && searchContent && contentResults.length === 0) {
    console.log(`[Indexing:searchKeyword] No results, trying fallback grep with regex mode`);
    try {
      const fallbackResults = await this.grepSearcher.grep(source, query, {
        regex: true,
        limit: limit * 2,
        maxMatchesPerFile: 1,
        caseSensitive: false,
        excludeExtensions: [...BINARY_EXTENSIONS]
      });
      console.log(`[Indexing:searchKeyword] Fallback results: ${fallbackResults.length}`);
      for (const r of fallbackResults) {
        if (!seen.has(r.path)) {
          seen.add(r.path);
          results.push({ path: r.path, rank: -0.3, contentMatches: { line: r.line, content: r.content } });
        }
      }
    } catch (err) {
      console.error(`[Indexing:searchKeyword] Fallback grep also failed:`, err.message);
    }
  }
  
  results = results.slice(0, limit);
  console.log(`[Indexing:searchKeyword] Final results: ${results.length}`);
  
  return { /* ... */ };
}
```

## Fix 8 — nLogger integration for better log management — 2026-04-05

**Problem:** Console logs are ephemeral and hard to search. No structured logging for production debugging.

**Solution:** Integrated [nLogger](https://github.com/herrbasan/nLogger.git) as a Git submodule for persistent, structured logging.

**Features:**
- **Session logs**: One file per server session with human-readable format
- **Main rolling log**: JSON Lines format for machine parsing, rotates by size (10MB default)
- **Automatic sanitization**: Truncates long strings, replaces base64 data with placeholders
- **Configurable retention**: Auto-deletes old logs (default: 7 days)
- **Buffered writes**: Better I/O performance with periodic flush

**Files Modified:**
1. `nLogger/` — Added as Git submodule
2. `src/utils/logger.js` — Wrapper that configures logger for nIndexer
3. `src/server.js` — Replaced `console.log` with `logger.info/warn/error`
4. `src/services/indexing.service.js` — Added logger import and usage
5. `src/config.js` — Added `logs.dir` configuration
6. `config.json` — Added logs configuration section
7. `.gitignore` — Added `logs/` directory

**Configuration:**
```json
{
  "logs": {
    "dir": "./logs",
    "retentionDays": 7
  }
}
```

**Log Output:**
```
logs/
├── 2026-04-05-03-52-35-nidx-l86z4a.log  # Session log (human-readable)
└── main-0.log                            # Rolling main log (JSON Lines)
```

**Session Log Format:**
```
[2026-04-05T03:52:35.123Z] [INFO] [Server] nIndexer service starting {"host":"localhost","port":3666,"sessionId":"nidx-l86z4a"}
[2026-04-05T03:52:36.456Z] [INFO] [WebSocket] Client connected: abc-123 from ::1
```

**Main Log Format (JSON Lines):**
```json
{"ts":"2026-04-05T03:52:35.123Z","level":"INFO","type":"Server","msg":"nIndexer service starting","meta":{"host":"localhost","port":3666},"session":"nidx-l86z4a"}
```

**Usage:**
```javascript
import { getLogger } from './utils/logger.js';

const logger = getLogger();

logger.info('Server started', { port: 3000 }, 'System');
logger.warn('High memory usage', { percent: 85 }, 'Monitoring');
logger.error('Database failed', error, { query: 'SELECT...' }, 'Database');
logger.debug('Verbose debug info', { details: '...' }, 'Debug');
```

**Querying Logs:**
```bash

# Filter by session

# View recent session log

```

4. Make some requests, then check main log: `cat logs/main-0.log | jq .`

**Verdict:** nLogger provides production-ready logging with minimal overhead. Session logs help debug specific server runs, while the main rolling log enables centralized log aggregation and analysis.
